1. PROJECT OVERVIEW
───────────────────────────────────────────────────────────────────────────────
A self-hosted, real-time chat application inspired by Rocket.Chat.
Runs as three Docker containers (MongoDB · .NET 8 backend · Nginx) on an Unraid
server, also accessible externally via Cloudflare Tunnel.

Features already implemented (do not re-implement):
  • Multi-room public and private channels
  • Direct messages (DMs) between two users
  • Real-time messaging via SignalR
  • File attachments (images + audio) stored in GridFS
  • User avatars stored in GridFS
  • Emoji reactions on messages
  • Message editing and soft-delete (own messages only, enforced server-side)
  • Message pinning (admin / private-channel members)
  • Message search within a room
  • Paginated message history (cursor-based, configurable limit)
  • Link/OG preview (server-side fetch, SSRF-protected)
  • Presence (online/offline) tracking
  • Typing indicators
  • JWT-based authentication stored in HttpOnly cookies
  • CSRF double-submit cookie protection
  • Mandatory TOTP (Time-based One-Time Password / 2FA) for all users and admin
  • Per-user account enable/disable (admin)
  • Push notifications via Web Push / VAPID
  • Admin panel (separate admin.html, admin JWT role)
  • Configurable message auto-purge (background service)
  • Configurable history-display limit

Deployment URL : http://<YOUR_SERVER_IP>:<PORT>  (also via Cloudflare Tunnel)
Repository root: dashboard-app/

5. AUTHENTICATION & 2FA FLOW
───────────────────────────────────────────────────────────────────────────────
Self-registration is PERMANENTLY DISABLED.  Only admin creates accounts.

── Regular user login (TOTP is mandatory for all users) ─────────────────────

Step 1 — Password check  POST /api/auth/login  { username, password }
  • If user has not yet configured 2FA (TotpEnabled = false):
      → 200  { requiresTotpSetup: true, tempToken: "<15-min setup JWT>" }
      The frontend must direct them to the first-time 2FA setup flow.
  • If 2FA is configured:
      → 200  { requiresTOTP: true, tempToken: "<5-min pending JWT>" }
      The frontend must prompt for the 6-digit authenticator code.

Step 2a — First-time 2FA setup (new accounts or reset)
  POST /api/auth/setup-totp-first  { tempToken }
      → 200  { secret, uri }   (uri is an otpauth:// deep-link / QR code source)
  POST /api/auth/confirm-totp-first  { tempToken, code }
      → 200  { token, username, displayName, userId, avatarColor, avatarFileId,
               totpEnabled: true }
      Sets full-session HttpOnly JWT cookies.

Step 2b — Normal TOTP verification
  POST /api/auth/verify-totp  { tempToken, code }
      → 200  { token, username, displayName, userId, avatarColor, avatarFileId,
               totpEnabled }
      Sets full-session HttpOnly JWT cookies.

── Admin login (admin.html) ──────────────────────────────────────────────────
Step 1  POST /api/admin/login  { username, password }
  • If ADMIN_TOTP_SECRET env var is not set:
      → 200  { token, username }   (immediate full admin JWT in cookies)
  • If ADMIN_TOTP_SECRET is set:
      → 200  { requiresTOTP: true, tempToken: "<5-min pending JWT>" }

Step 2  POST /api/admin/verify-totp  { tempToken, code }
      → 200  { token, username }   (full admin JWT in cookies)

── Managing 2FA on an authenticated account ─────────────────────────────────
  POST /api/users/2fa/setup      — generate new TOTP secret (pending confirmation)
  POST /api/users/2fa/confirm    — { code } — activate the pending secret
  POST /api/users/2fa/disable    — { password } — disable 2FA (requires password re-entry)
  GET  /api/admin/setup-totp     — admin: generate new TOTP secret for admin account
                                   (requires existing admin JWT — used to rotate the secret)

── Credentials & storage ─────────────────────────────────────────────────────
  Admin credentials : ADMIN_USERNAME, ADMIN_PASSWORD in .env
  Admin TOTP secret : ADMIN_TOTP_SECRET in .env (optional; absent = admin TOTP disabled)
  Backend hashes ADMIN_PASSWORD with BCrypt at startup.
  Admin is NOT a MongoDB document; a synthetic User object is constructed for JWT only.
  User TotpSecret and TotpEnabled are stored on the User document in MongoDB.

── JWT settings ──────────────────────────────────────────────────────────────
  Key        : Jwt__Key env var  (32+ char HMAC-SHA256 secret)
  Issuer     : ChatApp
  Audience   : ChatAppUsers
  Expiry     : 24 hours (configurable via Jwt:ExpirationHours in appsettings.json)

── Token types (role claims) ─────────────────────────────────────────────────
  "user"         — full chat access; set on successful login completion
  "admin"        — admin panel access; set after admin login completion
  "totp_pending" — 5-min short-lived; used while awaiting TOTP code input
  "totp_setup"   — 15-min short-lived; used during first-time 2FA setup

── Token transport ───────────────────────────────────────────────────────────
  Primary   : HttpOnly "chatToken" cookie (set by SetAuthCookies helper)
  SignalR   : ?access_token=<token> query string (JwtBearerEvents.OnMessageReceived)
  CSRF      : Separate "csrfToken" readable cookie + X-CSRF-Token header checks
              (double-submit pattern — see §14 Security)

── Authorization policies ────────────────────────────────────────────────────
  .RequireAuthorization()          — any JWT with role "user" or "admin"
  .RequireAuthorization("AdminPolicy") — role must be "admin"
  Token validation: issuer ✓, audience ✓, lifetime ✓, signing key ✓

── Session restore ───────────────────────────────────────────────────────────
  GET /api/auth/me  (RequireAuthorization)
  Returns a fresh JWT + user profile from the existing HttpOnly cookie.
  Called on page load when localStorage has profile data but the in-memory
  token is gone (e.g. after a tab refresh).

── Logout ────────────────────────────────────────────────────────────────────
  POST /api/auth/logout — clears both "chatToken" and "csrfToken" cookies.

── Password / username rules ─────────────────────────────────────────────────
  username : ≥ 3 characters
  password : ≥ 6 characters
  BCrypt work factor : 11

── ValidateTempToken helper (Program.cs local function) ─────────────────────
  Validates a short-lived JWT and verifies its role claim matches the expected
  value (e.g. "totp_pending" or "totp_setup").  Returns ClaimsPrincipal or null.
  Used by all TOTP verification endpoints.

6. RATE LIMITING
───────────────────────────────────────────────────────────────────────────────
── HTTP middleware (AddRateLimiter — .NET 8 built-in) ───────────────────────
  "login"    — FixedWindow: 10 req/min  → all login + TOTP endpoints
  "messages" — FixedWindow: 5 req/s    → /api/rooms/{id}/upload

  Rejection: HTTP 429 Too Many Requests
  Apply with: .RequireRateLimiting("login") or .RequireRateLimiting("messages")

── SignalR per-user rate limiting (ChatHub.cs in-memory) ────────────────────
  SendMessage     : max 5 messages per second per user
  EditMessage     : max 3 edits per second per user
  ToggleReaction  : max 5 reactions per second per user
  SendTyping      : max 2 typing signals per second per user

  Implementation: ConcurrentDictionary<userId, (windowStart, count)> per action.
  Exceeded limits silently drop typing/send, or emit RateLimited to caller.
  Stale entries are cleaned up every 5 minutes by a static Timer.

  Auth / Session:
  GET   /api/auth/me                  — restore session: returns fresh JWT + profile
  POST  /api/auth/logout              — clear auth cookies

  Rooms:
  GET   /api/rooms                    — list accessible rooms (admins see all)
  POST  /api/rooms                    — create room { name, description, isPrivate }

                                        (caller must be a member or admin)

  GET   /api/rooms/{roomId}/messages          — paginated history, oldest-first

                                                (own message only; broadcasts MessageEdited)

                                                      (broadcasts ReactionsUpdated)
  GET   /api/rooms/{roomId}/messages/search   — full-text search ?q=<query>&limit=<n>

9. SIGNALR HUB  (ChatHub.cs)
───────────────────────────────────────────────────────────────────────────────
URL: /chathub   — decorated with [Authorize] (any valid JWT)

── Server methods (client calls these) ──────────────────────────────────────
  JoinRoom(roomId)
      Adds caller to SignalR group.  Validates private-channel membership
      and DM participation server-side.  Broadcasts UserJoined.

  LeaveRoom(roomId)
      Removes caller from group.  Broadcasts UserLeft.

  JoinDm(otherUserId)
      Derives deterministic DM room ID ("dm_<smallerId>_<largerId>").
      Adds caller AND all active connections of the other user to the group.
      Sends DmInvite event to the other user's connections.

  SetActiveRoom(roomId?)
      Records which room the connection is currently viewing.  Used to
      skip push notifications for users who already see the message live.

  SendTyping(roomId)
      Broadcasts UserTyping to others in the group.
      Per-user rate-limited (2/s); silently dropped if exceeded.

  SendMessage(roomId, content, clientMsgId?)
      Validates access.  Persists to MongoDB.  Broadcasts ReceiveMessage
      to the group.  clientMsgId is echoed back for optimistic UI confirmation.
      Fires push notifications to offline / not-viewing users (async, fire-and-forget).
      Per-user rate-limited (5/s).

  EditMessage(roomId, messageId, newContent)
      Verifies ownership and roomId match.  Persists edit.
      Broadcasts MessageEdited.  Per-user rate-limited (3/s).

  DeleteMessage(roomId, messageId)
      Verifies ownership and roomId match.  Soft-deletes.
      Broadcasts MessageDeleted.

  ToggleReaction(roomId, messageId, emoji)
      Toggles userId in the emoji bucket.  Broadcasts ReactionsUpdated.
      Per-user rate-limited (5/s).

── Client events (server sends these) ───────────────────────────────────────
  OnlineUsers([{ userId, username }])
      Sent to caller immediately on connect — full snapshot of online users.

  UserOnline(userId, username)
      Broadcast to all when a user opens a new connection.

  UserOffline(userId, username)
      Broadcast to all only when a user's LAST connection disconnects
      (multi-tab safe — uses ConcurrentDictionary keyed by connectionId).

  UserJoined(username, roomId)
  UserLeft(username, roomId)
  UserTyping(username, roomId)

  ReceiveMessage({ id, roomId, senderId, senderUsername, senderDisplayName,
                   senderAvatarColor, senderAvatarFileId, content, deleted,
                   editedAt, reactions, attachmentId, attachmentName,
                   attachmentType, sentAt, clientMsgId? })

  MessageEdited({ messageId, roomId, content, editedAt })
  MessageDeleted({ messageId, roomId })
  ReactionsUpdated({ messageId, roomId, reactions })
  MessagePinned({ messageId, roomId, isPinned })
  RoomUpdated({ roomId, name, description })

  DmInvite(dmRoomId)
      Sent to the recipient when someone initiates a DM with them so their
      client can join the SignalR group without polling.

  RateLimited(message)
      Sent to the caller when SendMessage or EditMessage exceeds the per-user limit.

  ForceLogout(message)
      Sent to all connections of a user when an admin disables their account.
      The client must clear auth state and redirect to the login screen.

── Presence tracking ─────────────────────────────────────────────────────────
  Static ConcurrentDictionary<connectionId, (UserId, Username)>  _online
  Static ConcurrentDictionary<connectionId, roomId>              _activeRoom
  Static ConcurrentDictionary<connectionId, DateTime>            _connectionTime

  A periodic Timer runs every 5 minutes to purge stale rate-limit and orphaned
  connection entries (guards against any memory leak from missed disconnect events).

── EnrichMessages helper (Program.cs local function) ─────────────────────────
  Called whenever message history is returned from the API.
  Looks up each unique senderId in the Users collection and overwrites
  SenderAvatarFileId, SenderAvatarColor, and SenderDisplayName with current values.
  This ensures old messages (stored before a user updated their avatar) still render
  with the latest avatar.

14. SECURITY
───────────────────────────────────────────────────────────────────────────────
All of the following are already implemented:

HTTP Security Headers (middleware in Program.cs, applied to all responses):
  X-Frame-Options              : SAMEORIGIN
  X-Content-Type-Options       : nosniff
  Strict-Transport-Security    : max-age=31536000; includeSubDomains
  Referrer-Policy              : strict-origin-when-cross-origin
  Permissions-Policy           : camera=(), geolocation=(), payment=()
  Content-Security-Policy      : default-src 'self'; script-src 'self'
    https://cdnjs.cloudflare.com; style-src 'self' 'unsafe-inline'
    https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com;
    img-src 'self' blob: data:; media-src 'self' blob:;
    connect-src 'self' wss: ws: https://cdnjs.cloudflare.com;
    object-src 'none'; base-uri 'self'; frame-ancestors 'self'

CSRF Protection (double-submit cookie pattern):
  On login, SetAuthCookies sets both "chatToken" (HttpOnly) and "csrfToken"
  (readable by JS).  All state-changing HTTP requests (POST/PUT/PATCH/DELETE)
  must send the csrfToken value in an X-CSRF-Token header.  The middleware
  compares cookie vs header and returns 403 on mismatch.
  Exemptions: /api/auth/*, /api/admin/login, /api/admin/verify-totp, /chathub.
  apiFetch() in app.js reads the cookie and sends the header automatically.

Magic-bytes validation (HasValidMagicBytes helper in Program.cs):
  Both upload endpoints (chat attachments and user avatars) read the first
  12 bytes of the uploaded stream and verify they match the claimed MIME type.
  Prevents attackers from uploading HTML/executables with a spoofed Content-Type.
  Supported: JPEG, PNG, GIF, WebP, OGG, WAV, MP3/MPEG, MP4/M4A, WebM.

XSS Prevention:
  ALL user-supplied content must pass through escapeHtml() before being
  inserted into the DOM.  No exceptions.  innerHTML with raw user data is
  forbidden; use textContent or escapeHtml().

Server-side auth enforcement:
  Edit/delete/pin ownership enforced in MessageRepository and ChatHub (senderId check).
  Private channel history access checked on every GET /messages call.
  Private room join (SignalR JoinRoom) validates membership server-side.
  DM access verified by checking the caller's userId is in the room ID.
  Frontend UI guards (hiding buttons, etc.) are cosmetic only — not security.

Push endpoint validation:
  POST /api/push/subscribe validates the endpoint is an HTTPS URL belonging to
  a known push service domain (Mozilla, Windows, Google, Apple).

SSRF protection in /api/og:
  Resolves DNS before fetching; rejects private/loopback/link-local IP ranges.
  Pins the resolved IP address for the actual request to prevent DNS rebinding.
  Limits HTML to 64 KB and caps regex execution to 2 seconds.

File serving:
  GET /api/files/{fileId} requires a valid JWT.
  Only known-safe media types (images + audio) are served inline.
  All other types force a download with a sanitised filename.

Disabled accounts:
  Disabled users get a 403 on /api/auth/me and /api/auth/login.
  When an admin disables a user, ForceLogout is sent to all their active
  SignalR connections so the session terminates immediately.

Other:
  Backend runs as non-root Linux user (appuser:appgroup) inside the container.
  MongoDB is not exposed on the host — only on the internal Docker network.
  Rotate secrets: edit .env → docker compose up -d backend.

16. CONVENTIONS FOR FUTURE LLM SESSIONS
───────────────────────────────────────────────────────────────────────────────
These rules apply to ALL code changes in this repository.

── Architecture rules ────────────────────────────────────────────────────────
  • .NET 8 Minimal API only — no Controllers, no Startup.cs
  • All HTTP endpoints live in Program.cs
  • New MongoDB collection:
      1. Add Model class in backend/Models/
      2. Add IRepository interface in backend/Repositories/
      3. Add Repository implementation
      4. Register AddSingleton in Program.cs (see existing pattern)
  • New SignalR server→client event:
      1. Add method call in ChatHub.cs (or hub context in Program.cs)
      2. Add handler in connectSignalR() in app.js
  • New client-side state variables: declare at top of app.js with other globals
  • Rate-limit any endpoint that is write-heavy or a brute-force target:
      .RequireRateLimiting("login") or .RequireRateLimiting("messages")
      or add a per-user bucket in ChatHub for SignalR methods

── Immutable constraints ─────────────────────────────────────────────────────
  | Constraint                  | Rule                                           |
  |-----------------------------|------------------------------------------------|
  | MongoDB connection          | ALWAYS MongoClientSettings + MongoCredential   |
  |                             | NEVER a connection string URL                  |
  | Docker images               | ALWAYS Linux-only base images                  |
  | Frontend                    | NO build step — plain HTML/CSS/Vanilla JS only |
  | Self-registration           | MUST remain disabled — admin-only creation     |
  | Admin storage               | NEVER in MongoDB — synthetic User for JWT only |
  | File storage                | GridFS (already integrated) — use it           |
  | DOM output                  | ALL user content via escapeHtml() — no exceptions |
  | Auth enforcement            | Ownership checks + private room access MUST be |
  |                             | server-side only; frontend guards are cosmetic  |
  | 2FA for users               | TOTP is mandatory — do not add a bypass path   |

── Secret management ─────────────────────────────────────────────────────────
  appsettings.json    — non-secret defaults only (host, issuer, expiry, etc.)
  Production secrets  — Docker env vars injected via .env file
  AuthService.GenerateToken(user, role):
    role = "user"  for chat users
    role = "admin" for admin

── DM room ID convention ─────────────────────────────────────────────────────
  dm_<smallerId>_<largerId>   (string-sort both user IDs, smaller first)
  This makes the room ID deterministic regardless of who initiates.
  DM rooms have NO ChatRoom document in MongoDB — access is controlled purely
  by whether the userId appears in the room ID.  Messages in DMs use this as
  their RoomId like any other message.

── Message pagination convention ─────────────────────────────────────────────
  GET /api/rooms/{roomId}/messages?before=<msgId>
  Omitting `before` returns the latest N messages (N = ChatSettings.HistoryLimit).
  Providing a msgId returns messages older than that message (for scroll-up loading).

── TOTP integration notes ────────────────────────────────────────────────────
  Library: OtpNet (Base32Encoding, KeyGeneration, Totp classes)
  Verification uses VerificationWindow.RfcSpecifiedNetworkDelay (±1 step tolerance)
  TOTP secret: 20 random bytes encoded as Base32 string, stored in User.TotpSecret
  Admin TOTP: Base32 secret stored in ADMIN_TOTP_SECRET env var (not in MongoDB)

── Push notification notes ───────────────────────────────────────────────────
  Library: WebPush (VapidDetails, PushSubscription, WebPushClient)
  Keys generated once with `npx web-push generate-vapid-keys`
  Push is fire-and-forget from SendMessage; push is skipped if:
    - VAPID keys are not configured (start with "CHANGE_ME")
    - Recipient is the sender
    - Recipient is currently viewing the same room (tracked via _activeRoom)
  Expired subscriptions (HTTP 410/404) are deleted automatically.
