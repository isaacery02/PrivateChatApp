1. PROJECT OVERVIEW
───────────────────────────────────────────────────────────────────────────────
A real-time, multi-room chat application inspired by Rocket.Chat.
Deployed as three Docker containers on an Unraid server.

Deployment URL : http://<YOUR_SERVER_IP>:<PORT>  (also accessible via Cloudflare Tunnel)

4. MONGODB CONFIGURATION
───────────────────────────────────────────────────────────────────────────────
Host container name : mongodb          (Docker internal DNS)
Port               : 27017
Application DB     : chat_db
Auth Source DB     : my_app_db
App username       : app_user
App password       : <set MONGO_APP_PASSWORD in .env — never commit the real value>

CRITICAL: The password contains literal "@" characters.
The backend MUST use MongoClientSettings + MongoCredential — NOT a connection
string URL.  This must never be changed to a connection string.

Collections:
  users      — registered user accounts
  messages   — all chat messages  (index: roomId + sentAt DESC)
  chat_rooms — channel definitions (index: name UNIQUE)
  fs.files / fs.chunks — GridFS buckets for images and avatars

Seeded default rooms: #general, #random  (via mongo-init.js)

GridFS buckets (stored inside the mongodb container, no extra storage needed):
  Chat image uploads  → POST /api/rooms/{roomId}/upload  (≤10 MB)
  User avatar uploads → POST /api/users/avatar           (≤2 MB)
  Download endpoint   → GET  /api/files/{fileId}         (authenticated)

5. AUTHENTICATION (JWT)
───────────────────────────────────────────────────────────────────────────────
Self-registration is PERMANENTLY DISABLED. Only admin creates accounts.

Regular user login:
  POST /api/auth/login  { username, password }
  → { token, username, displayName, userId, avatarColor, avatarFileId }
  Token role claim: "user"
  Rate-limited: "login" policy (10 req/min)

Admin login (admin.html only):
  POST /api/admin/login  { username, password }
  → { token, username }
  Token role claim: "admin"
  Rate-limited: "login" policy (10 req/min)

Admin credential storage:
  File  : .env  (git-ignored)
  Keys  : ADMIN_USERNAME, ADMIN_PASSWORD
  Backend hashes ADMIN_PASSWORD with BCrypt at startup.
  Admin is NOT a MongoDB document; synthetic User object is used for token only.

JWT settings (appsettings.json → "Jwt", override via env vars in prod):
  Key        : Jwt__Key env var  (32+ char HMAC-SHA256 secret)
  Issuer     : ChatApp
  Audience   : ChatAppUsers
  Expiry     : 24 hours

Token transport:
  REST      → Authorization: Bearer <token>
  SignalR   → ?access_token=<token>  (query string, JwtBearerEvents.OnMessageReceived)

Authorization policies:
  (default)      — any valid JWT    → all /api/* except login; SignalR hub
  "AdminPolicy"  — role == "admin"  → all /api/admin/* endpoints

Password rules: username ≥ 3 chars, password ≥ 6 chars, BCrypt rounds = 11

───────────────────────────────────────────────────────────────────────────────
6. RATE LIMITING (.NET 8 AddRateLimiter)
───────────────────────────────────────────────────────────────────────────────
"login"    — FixedWindow: 10 requests per minute  → applied to both login endpoints
"messages" — FixedWindow: 5 requests per second   → applied to /upload endpoint
Rejection: HTTP 429 Too Many Requests

8. API ENDPOINTS
───────────────────────────────────────────────────────────────────────────────
Anonymous:
  POST /api/auth/login               — user login  [rate: login]
  POST /api/admin/login              — admin login [rate: login]

Admin JWT required:
  GET    /api/admin/users            — list users (id, username, displayName,
                                       avatarColor, avatarFileId, createdAt)
  POST   /api/admin/users            — create user { username, password }
  DELETE /api/admin/users/{id}       — delete user

Any valid JWT:
  GET  /api/rooms                    — accessible rooms (public + user's private)
  POST /api/rooms                    — create room { name, description, isPrivate }
  POST /api/rooms/{id}/invite        — invite user to private channel { userId }
                                       (caller must be a member or admin)

  GET  /api/rooms/{id}/messages      — last 50 messages, oldest-first
                                       (private channels: access guarded)
  PUT  /api/rooms/{id}/messages/{msgId}   — edit message { content }
                                           (own messages only; broadcasts MessageEdited)
  DELETE /api/rooms/{id}/messages/{msgId} — soft-delete message
                                           (own messages only; broadcasts MessageDeleted)
  POST /api/rooms/{id}/messages/{msgId}/react  — toggle emoji { emoji }
                                                 (broadcasts ReactionsUpdated)

  POST /api/rooms/{id}/upload        — upload image to GridFS (multipart)
                                       allowed: JPEG/PNG/GIF/WebP ≤10 MB
                                       [rate: messages]
  GET  /api/files/{fileId}           — download file from GridFS (as blob)

  PATCH /api/users/me                — update own displayName / avatarColor
  POST  /api/users/avatar            — upload own avatar (JPEG/PNG/WebP ≤2 MB)

Anonymous:
  GET  /health                       — { status, timestamp }

9. SIGNALR HUB  (ChatHub.cs)
───────────────────────────────────────────────────────────────────────────────
URL: /chathub   — [Authorize] attribute (any valid JWT)

Server methods (client invokes these):
  JoinRoom(roomId)                   → joined group; broadcasts UserJoined
  LeaveRoom(roomId)                  → left group;   broadcasts UserLeft
  SendTyping(roomId)                 → broadcasts UserTyping to others in group
  SendMessage(roomId, content)       → persists to MongoDB; broadcasts ReceiveMessage
  EditMessage(roomId, msgId, text)   → updates MongoDB (own only); broadcasts MessageEdited
  DeleteMessage(roomId, msgId)       → soft-deletes (own only); broadcasts MessageDeleted
  ToggleReaction(roomId, msgId, emoji) → toggles reaction; broadcasts ReactionsUpdated

Client events (server sends these):
  OnlineUsers([{userId, username}])  → initial snapshot on connect
  UserOnline(userId, username)       → presence: user connected
  UserOffline(userId, username)      → presence: user has no more connections
  UserJoined(username, roomId)
  UserLeft(username, roomId)
  UserTyping(username, roomId)
  ReceiveMessage({ id, roomId, senderId, senderUsername, senderDisplayName,
                   senderAvatarColor, senderAvatarFileId, content, deleted,
                   editedAt, reactions, attachmentId, attachmentName,
                   attachmentType, sentAt })
  MessageEdited({ messageId, roomId, content, editedAt })
  MessageDeleted({ messageId, roomId })
  ReactionsUpdated({ messageId, roomId, reactions })

Presence tracking:
  Static ConcurrentDictionary<string,(string UserId, string Username)>
  keyed by connectionId.  UserOffline only fires when the last connection
  for a userId disconnects (multi-tab safe).

14. SECURITY NOTES
───────────────────────────────────────────────────────────────────────────────
- Self-registration permanently disabled; admin-only account creation
- Admin credentials live only in .env and Docker env vars — never in MongoDB
- Admin is NOT a MongoDB document; synthetic User object used for JWT only
- All user-supplied content passes through escapeHtml() before DOM insertion
- JWT Bearer validates issuer, audience, lifetime, and signing key
- Rate limiting on login (brute-force protection) and upload endpoints
- EditAsync / SoftDeleteAsync enforce senderId ownership — backend-side check
  (not just frontend; cannot delete/edit another user's message even via direct API call)
- Private channel history and membership guarded server-side (not just hidden in UI)
- GridFS files served authenticated-only (GET /api/files/* requires valid JWT)
- Backend runs as non-root Linux user (appuser:appgroup) inside the container
- MongoDB is not exposed on the host; only on internal Docker network
- Rotate secrets: edit .env, then `docker compose up -d backend`

15. DEPLOYMENT
───────────────────────────────────────────────────────────────────────────────

  cd dashboard-app
  docker compose up --build -d

  docker compose up --build -d backend   # backend only, faster
  docker compose up --build -d           # full rebuild

  docker compose logs -f
  docker compose logs -f backend

Stop:
  docker compose down

Reset database (WARNING — destroys all data including uploaded files):
  docker compose down -v

  cd dashboard-app/backend
  dotnet build -c Release

16. CONVENTIONS FOR FUTURE LLM SESSIONS
───────────────────────────────────────────────────────────────────────────────
Code style:
  - .NET 8 Minimal API only — no Controllers, no Startup.cs
  - New collections → add Repository interface + impl, register AddSingleton in Program.cs
  - New SignalR server events → add method to ChatHub.cs, handle in connectSignalR() in app.js
  - New client-side state → declare at top of app.js beside other state globals
  - Rate-limit sensitive new endpoints with .RequireRateLimiting(...)

Immutable constraints:
  - MongoDB: ALWAYS use MongoClientSettings + MongoCredential (password contains @)
  - Docker base images: ALWAYS Linux-only (no windows/* images)
  - Frontend: NO build pipeline — plain HTML/CSS/Vanilla JS only
  - Self-registration: MUST remain disabled — user creation is admin-only
  - Admin: NEVER stored in MongoDB; always synthetic User object for JWT
  - GridFS: already integrated; use it for any binary/file storage
  - All DOM output: MUST pass through escapeHtml() to prevent XSS
  - Auth checks (edit/delete ownership, private room access): server-side only;
    never rely on frontend-only guards

Secret management:
  - appsettings.json: non-secret defaults only
  - Production secrets: Docker env vars from .env file
  - GenerateToken(user, role): pass "user" for chat users, "admin" for admin
